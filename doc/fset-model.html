Hemlock — FSet Buffer Model

How immutable state fits into a mutable editor without rewriting commands.

1. The problem with mutable buffers

Hemlock has the same model. Lines are a doubly-linked list. Marks are pointers into lines. insert-string mutates the line, adjusts the gap, fixes up marks. If anything reads a buffer while a command is halfway through a multi-line insert, it sees garbage. This is why serve-event serializes everything. It's not a design flaw — it's the only correct behavior given mutable state.

6. What FSet is, concretely

buffer-state struct (src/text/fset-state.lisp)
  • lines — FSet seq of strings. One string per line. Balanced tree internally. Insert/delete at any index: O(log n). Lookup by index: O(log n).
  • marks — FSet map from mark-id to (line-idx charpos kind). Every edit produces new marks with adjusted positions.
  • meta — FSet map of buffer properties (name, modes, modified-tick, etc.).
  • tick — monotonically increasing integer. Bumped on every edit. Consumers compare ticks to detect staleness.
structural sharing
  • Editing line 50 of a 10,000-line buffer copies ~13 tree nodes (log2 10000). The other 9,999 lines are shared between old and new state.
  • A snapshot is just a pointer to the root. Cost: one pointer copy.
  • Old states are garbage collected when no one holds them. If undo holds 100 states and they share 99% of lines, memory cost is ~1% overhead per undo level.
buffer-snapshot struct (for display)
  • Lightweight view extracted from buffer-state via state->snapshot.
  • Contains: lines, line-count, point position, display-start, modes, modified-p, colors (from tree-sitter), modeline string, window dimensions.
  • Everything the display needs to render. No back-references to mutable state.
  • Display diffs old snapshot vs new snapshot to minimize terminal writes.

7. What this does NOT require

No changes to:
  • Commands — all 50+ command files untouched. insert-string, delete-characters, with-mark all work as-is.
  • Marks — still mutable pointers. Commands use them normally. FSet marks are a parallel data structure, not a replacement.
  • Gap buffer — still the write-path representation. Fast single-cursor edits, same as always.
  • Command loop — still single-threaded, still serialized. The command loop is the writer. It doesn't need to change.
  • Keybindings — nothing changes.
What changes:
  • buffer-modified-hook — gains a shadow function that rebuilds FSet state after each mutation. ~20 lines of code.
  • Display — reads from snapshots instead of directly walking the mutable line list. Existing tty/display.lisp adapts.
  • Tree-sitter — receives snapshots instead of reading mutable lines. Parses immutable data. Colors tagged with tick.
  • Undo — optionally backed by state history instead of the current reverse-edit approach. Simpler, more reliable.
The mutable model stays. It's the fast path for commands. FSet is the safe path for readers. They coexist. Commands write to mutable buffers. The shadow propagates to FSet. Readers consume FSet snapshots. Nobody waits for anybody. This is the dual-write architecture from archv2.html Phase 2 — and it's the realistic ceiling that gets you 90% of the parallelism benefits without rewriting the editor.

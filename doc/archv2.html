Hemlock — Architecture v2

Lessons from sento-remoting. The real path forward.

1. What sento-remoting proved

The branch attempted Option B (full actor model) in one shot. It partially worked. Here's what we learned.

Validated
  • Sento remoting works — two actor-systems communicating over localhost TCP. Smoke test proves ping, eval, fire-and-forget.
  • FSet immutable buffer state works — 15 tests passing. Insert, delete, region ops, mark adjustment, structural sharing. Pure functions, zero coupling to hemlock internals.
  • Actor protocols are sound — buffer, command, display, input as separate actors with deferred receive functions. Clean separation of protocol from implementation.
  • Agent registry works — uniform ask-s interface for local/process/remote agents.
  • hemlock.core.asd layering works — 10-layer module structure with clean dependency graph.
Failed
  • Display-actor was a toy — reimplemented terminal rendering from scratch. No dis-line diffing, no scroll optimization, no hunk management, no modeline fields. Regressed everything.
  • Input-actor duplicated code — copy-pasted bracketed paste, focus, SGR mouse, key event parsing from tty/input.lisp into tty/input-actor.lisp. Two copies of the same logic.
  • Tree-sitter broke — actor indirection between TS and buffers added latency and race conditions that didn't exist in the synchronous model.
  • Gray streams over actors — reimplemented old wire-backed stream pattern on top of sento. Unnecessary. Sento messages are the I/O.
Core lesson: The architecture is right. The execution was wrong. Don't rewrite working code from scratch. Adapt existing code to work within the actor model incrementally.

2. The tmux model

hemlock is a client that bootstraps its own core. Not a daemon you manage. Not a server you start separately. Like tmux: you run it, it works. The server is an implementation detail.

Key insight: The client bootstraps the core. Like tmux, the user never thinks about servers or daemons. hemlock just works. The client is in-process — it IS the process. Agents connect from outside. Multiple clients can attach to the same core (pair programming, mobile + desktop, etc.).

3. FSet: the data foundation

Immutable buffer state is what makes parallel actor access safe without locks. This is already built and tested (src/text/fset-state.lisp, 15 tests passing).

Why FSet matters
  • No locks: actors hold snapshots, not references to mutable state. No reader-writer contention. No deadlocks. No races.
  • Structural sharing: inserting a character copies O(log n) tree nodes, not O(n) line strings. Snapshots are nearly free.
  • Undo for free: keep a list of old buffer-states. Undo = point at previous state. No diff/patch/reverse machinery.
  • Tree-sitter safety: TS parses a snapshot. If the buffer changes while parsing, the snapshot is still valid — just stale. Parse completes, colors tagged with tick. If tick < current, discard or re-parse. No corruption.

5. Backend protocol

Backends are display clients. The protocol is the same for all of them. The existing TTY and WebUI code adapts to this protocol — no rewrite.

Display protocol (core → client)
  • :snapshot — buffer-snapshot struct. Lines, marks, colors, modeline, dimensions. Everything needed to render.
  • :clear — clear display.
  • :resize cols lines — terminal/window resized.
  • :message text — echo area / status.

Client receives these and renders however it wants. TTY: escape sequences. WebUI: DOM updates. Future GUI: draw calls.

Input protocol (client → core)
  • :key-event ke — translated key event.
  • :mouse-event type x y modifiers — click, scroll, drag.
  • :resize cols lines — client reports new size.
  • :paste text — bracketed paste content.
  • :focus state — gained/lost focus.

Client translates raw input into these messages. TTY: parse escape sequences. WebUI: JS event handlers. Same messages either way.

No duplication. The existing tty/input.lisp keeps parsing escape sequences. The existing tty/display.lisp keeps rendering. They just get wired to send/receive actor messages instead of going through serve-event fd-handlers. The sento-remoting branch rewrote both from scratch. That was the mistake.

6. What to keep from sento-remoting

Keep
  • src/text/fset-state.lisp + t/fset-state.lisp — immutable buffer state. Foundation for everything. 15 tests. Zero coupling.
  • hemlock.core.asd module layering — the 10-layer structure is the right dependency graph.
  • src/actor/system.lisp — actor-system start/stop, remoting enable/disable.
  • src/actor/registry.lisp — agent registry with uniform ask-s interface.
  • src/actor/local-agent.lisp — in-process eval agent.
  • t/remoting-smoke.lisp — proves sento remoting works end-to-end.
  • src/actor/agent.lisp — agent-side actor (receives :eval, replies with result). Drop the gray streams, keep the actor and connect entry point.
Discard
  • src/tty/display-actor.lisp — toy reimplementation of terminal rendering. Use real tty/display.lisp.
  • src/tty/input-actor.lisp — duplicated input parsing. Adapt existing tty/input.lisp instead.
  • src/user/actor-impl.lisp — the buffer-actor and command-actor receive functions that wrap mutable buffers in FSet snapshots on every call. Rebuild this properly when FSet is the primary data model, not a dual-write afterthought.
  • Gray stream classes in src/actor/agent.lisp — agent-output-stream, agent-input-stream. Sento messages are the I/O. No wrapper needed.
  • src/actor/buffer-actor.lisp, command-actor.lisp, display-actor.lisp, input-actor.lisp — protocol stubs with deferred receive. Replace with real implementations that integrate with existing hemlock code instead of stubbing around it.

7. Migration phases

Each phase produces a working editor. No big-bang rewrite. The key difference from architecture.html: we're not choosing between Option A and Option B. We're building Option B incrementally, and every intermediate state is a working editor.

Phase 1 is urgent — gets eval working, proves sento as transport. Phase 2 is the architecture — FSet becomes the read model, display and TS become snapshot consumers, commands stay mutable. There is no Phase 3. Phase 2 is the end state. One thread writes (commands), many threads read (display, TS, agents, webui). That's the concurrency model.

8. What this means for main branch

Merge strategy
  • Cherry-pick from sento-remoting: fset-state.lisp, t/fset-state.lisp, hemlock.core.asd (the layering, not all the actor deps), actor/system.lisp, actor/registry.lisp, actor/local-agent.lisp, t/remoting-smoke.lisp.
  • Don't merge: tty/display-actor.lisp, tty/input-actor.lisp, actor-impl.lisp, gray stream classes, the protocol stubs.
  • On main: start Phase 1 cleanly. Wire up sento for agents. Leave display, input, tree-sitter, command loop untouched.
The end state
  • hemlock is a self-assembling editor (tmux model).
  • Commands mutate buffers on a single thread (fast, proven, unchanged).
  • Every mutation shadows into FSet state via buffer-modified-hook.
  • Display, tree-sitter, agents, webui read FSet snapshots (parallel, race-free).
  • Agents connect via sento remoting (local, process, remote).
  • Undo is a stack of immutable states. O(1) swap.
  • Multiple display clients on the same session (TTY + WebUI).
  • No locks. No CRDTs. No command rewrites. Just one writer, many readers.
